Continued same day at the user's request.

## What shipped

# Day 3 — Transcription (batch, no diarization)

Continued same day at the user's request.

## Plan change: nodejs-whisper → prebuilt binaries

`nodejs-whisper` compiles whisper.cpp from source, and this machine has no `make`,
`cmake`, `gcc`, or MSVC. Rather than install a multi-GB build toolchain, switched to
**prebuilt Windows binaries**, confirmed with the user first:

- `whisper-cli.exe` + DLLs from the official whisper.cpp GitHub release
  (`ggml-org/whisper.cpp` v1.9.1, `whisper-bin-x64.zip`)
- `ggml-base.bin` — the multilingual base model (not `.en`-only), since Day 10's
  translation feature will need non-English speech recognized too
- `ffmpeg.exe` (BtbN's static LGPL build) — needed because whisper.cpp's bundled
  decoder reads flac/mp3/ogg/wav but not WebM/Opus, which is what the browser's
  `MediaRecorder` produces

All three are downloaded by `backend/scripts/setupSpeechTools.js` (`npm run
setup:speech`), idempotent, skips anything already present. They live in
`backend/bin/`, gitignored — large binaries don't belong in the repo, and this keeps
the "free" promise honest (nothing paid, just a one-time local download).

## What shipped

- `services/transcribeService.js`: converts the uploaded audio to 16kHz mono WAV via
  ffmpeg, then runs `whisper-cli.exe` against it, returns the transcript text.
- `ConversationSession` gets `transcript` and `transcriptStatus`
  (`none`/`processing`/`done`/`failed`) fields.
- `stopConversation` kicks off transcription **after** sending the stop response
  (fire-and-forget) rather than making the doctor wait on the Stop click —
  transcription of a multi-minute recording could take a while on CPU.
- Conversation page polls every 3s while any session shows `processing`, and
  renders the transcript once it's `done`.

## Verified

- Unit-style check: generated a synthetic WAV, ran it through `transcribeAudio`
  directly — pipeline (webm → wav → whisper-cli → cleanup) completes without error.
- Real accuracy check: used Windows' built-in SAPI text-to-speech to generate actual
  spoken audio ("Take one tablet of amoxicillin every eight hours for five days.
  Report any fever."), converted to webm, ran it through the real API end-to-end.
  Transcript came back as "Take one tablet of a Moxasillin every 8 hours for 5 days.
  Report any fever." — correct content, exactly the kind of medication-instruction
  sentence this feature exists to capture (whisper mixed up "amoxicillin" spelling,
  which is expected ASR behavior on an uncommon drug name, not a bug in the code).
- Drove the full flow through the actual UI: started a session via the real API,
  stopped it with the real speech clip, confirmed the transcript appeared under the
  session without a manual refresh (polling worked), no console errors.
- Deleted all test records/audio afterward.

# Day 4 — Speaker diarization

Continued same day (user wants to finish the whole project today; confirmed with them
that "Day N" stays an honest label for each real chunk of work, not a claim about
which calendar day it happened on — see the conversation, not repeating the reasoning
here).

## Plan change #2: pyannote.audio → MFCC clustering

Two real blockers, found by actually trying rather than assuming:

1. pyannote.audio's accurate pipeline is **gated on Hugging Face** — needs an account,
   accepted model terms, and an access token. That's a manual step only the user can
   do. Asked; they picked the no-login path to keep moving.
2. The obvious no-login alternative, **Resemblyzer**, turned out to depend on
   `webrtcvad`, which ships no prebuilt Windows wheel and needs a C compiler — the
   exact same wall hit on Day 3. `pip install --dry-run` didn't catch this (it doesn't
   actually attempt the native build), only a real `pip install` did. Lesson: dry-run
   is not a real compatibility test for source-only packages.

Landed on **MFCC feature clustering**: `librosa` extracts 20 MFCCs per Whisper segment
(mean+std → 40-dim feature vector), `scikit-learn`'s `AgglomerativeClustering` groups
segments into speakers. No neural model, no gated download, no compiler — every
dependency in `pyservices/requirements.txt` ships a prebuilt wheel (verified with a
real install this time, not a dry-run).

## What shipped

- `pyservices/diarize.py` — takes a WAV + a list of `{startMs, endMs}` segments,
  outputs which segments share a speaker (`{"speakers": [0, 1, 0, ...]}`).
- `backend/scripts/setupDiarizeTools.js` (`npm run setup:diarize`) — creates the venv,
  installs `librosa` + `scikit-learn`.
- `services/diarizeService.js` — Node wrapper that shells out to the venv's python.
  Returns `null` (not a throw) if the tooling isn't installed, so a missing setup step
  degrades to "everything is Speaker 1" instead of failing the whole transcript.
- `transcribeService.js` reworked to return **timestamped segments** (whisper.cpp's
  `-oj` JSON output) instead of one flat string, so diarization has per-segment
  boundaries to work with.
- `ConversationSession.segments` — `[{startMs, endMs, text, speaker}]`. `transcript`
  (flat string) is still populated too, for the fallback render path.
- Conversation page now renders each segment as `Speaker N: <text>` instead of one
  undifferentiated paragraph.

## Bug found and fixed: nodemon killed transcription mid-flight

`nodemon` (used by `npm run dev`) watches `.json` files by default. Whisper's `-oj`
output writes a `.json` file into `backend/storage/audio/`, which is inside nodemon's
watched tree — so starting transcription would trigger nodemon to restart the server,
killing the fire-and-forget job before it could save results. Was invisible on Day 3
(that used `-otxt`, a `.txt` file, not a watched extension) and only surfaced today
after switching to JSON output. Fixed with `backend/nodemon.json` ignoring `storage/`,
`bin/`, `pyservices/venv/`, `uploads/`. Would have silently broken every real
Stop-with-audio call in dev before this fix.

## Verified

- Built a genuine two-voice test conversation using Windows SAPI's two distinct
  built-in voices (David, Zira) in a doctor→doctor→patient turn pattern via PowerShell,
  concatenated with ffmpeg, encoded to webm (matching real browser output).
- Ran it through the real API end-to-end: diarization returned `{"speakers": [0, 0, 1]}`
  — correctly grouping the two David segments together and separating the Zira segment,
  exactly matching ground truth.
- Confirmed the same result rendering correctly in the actual UI: "Speaker 1: Good
  morning. What symptoms are you experiencing?" / "Speaker 2: I have a sore throat and
  body ache since two days." / "Speaker 1: Take one tablet of paracetamol twice daily
  after food for five days." No console errors.
- This is also where the nodemon bug was caught — the first two attempts silently
  never finished processing; checking server logs (not just the API response) found
  the restart loop.
- Deleted all test records/audio afterward.

# Day 6 — Excel export

Continued same day at the user's request.

## Design choice: generate on download, not on Stop

The plan said "generate on Stop," but doing that would bake in whatever speaker
labels existed *at that moment* — and Day 5 lets the doctor keep relabeling speakers
after the fact. Generating the workbook fresh each time `GET .../excel` is called
means it always reflects the current `speakerRoles` mapping, and avoids storing a
second derived file alongside the audio. `exceljs` is fast enough for a
few-hundred-row transcript that this costs nothing noticeable per download.

## What shipped

- `services/excelService.js` — builds a workbook: patient name + date header, then a
  `Time | Speaker | Statement` table. Time is elapsed `mm:ss` from the start of the
  recording (not wall-clock), matching how transcript timestamps are normally read.
  Speaker resolves through `speakerRoles` when set, generic label otherwise.
  `exceljs` needed no native build step (pure JS, npm install just worked).
- `GET /api/conversations/:id/excel` — doctor-auth-gated, ownership-checked, 404s if
  there's no transcript yet. Streams the workbook directly to the response with
  `Content-Disposition: attachment`.
- Frontend: fetches the file as a blob (a plain `<a href>` can't carry the auth
  header) and triggers the download via a temporary anchor element.
- "⬇ Download Excel" button next to the transcript.

## Verified

- Ran the full pipeline (record → transcribe → diarize → relabel speakers to
  Doctor/Patient) via the real API, then fetched the .xlsx and **parsed it back with
  exceljs** (not just checked headers/size) to confirm actual cell contents:
  correct patient name, date, and all three rows with the right elapsed time,
  resolved role, and exact statement text — including the medication instruction
  ("Take one tablet of paracetamol twice daily after food for five days.") that's
  the whole point of this sheet.
- Confirmed a request with no token gets 401 (ownership/auth gate intact).
- Confirmed the click-to-download flow through the actual browser UI: clicking the
  button fired the real authenticated request and got a 200 with a binary payload

# Day 7 — Consent capture and audit log

Continued same day at the user's request.

## What shipped

- Consent checkbox appears once a patient is found; "Start recording" stays disabled
  until it's checked. Purely client-side gating isn't trusted, though — `POST
  /api/conversations` rejects with 400 unless `consentGiven === true` arrives in the
  body, and only then does `session.consent = { given: true, at: <timestamp> }` get
  set. A recording session can never exist in the database without recorded consent.
- `AuditLog` model — append-only (nothing in the app updates or deletes an entry):
  `{ doctorId, sessionId, action, at }`, `action` one of `start` / `stop` /
  `download-audio` / `download-excel`. An entry gets written at each of those four
  points in `conversationController.js`.
- `GET /api/conversations/:id/audit` — doctor-auth-gated, ownership-checked, returns
  a session's log entries in chronological order. No dedicated UI for it yet (not in
  the Day 7 scope), but it's there for verification and for whatever surfaces it
  later (Day 9's dashboard, most likely).

## Verified

  with no `consentGiven` field, and again with `consentGiven: false`, both correctly
  400 with "Patient consent is required before recording can start."

# Day 8 — Encryption at rest, and a real credential leak

Continued same day at the user's request.

## Real finding: leaked live credentials in a public repo

Before touching encryption, found that `backend/.env.example` — a tracked, committed
file, not gitignored — contained the **real** MongoDB Atlas password, `JWT_SECRET`,
and `ADMIN_PASSKEY`, not placeholders. Confirmed via GitHub's API that
`manish-g-p/carepulse-mern` is a **public** repo, and the leak has been there since
the initial commit. This is more severe than it sounds: a leaked `JWT_SECRET`
specifically means anyone could forge a valid doctor/admin token and walk straight
past every ownership check built across Days 1–7, since those checks all trust
`req.auth.doctorId` from the JWT.

Flagged this to the user immediately rather than silently patching around it, since
fixing the file doesn't undo the exposure — only rotating the actual credentials
does, and that requires their MongoDB Atlas account (something I can't do). They're
rotating the Atlas password themselves; I generated fresh random `JWT_SECRET` and
`ADMIN_PASSKEY` values (the user approved this — note this invalidates existing
doctor/admin sessions, which is fine, only test accounts exist) and replaced the
live values in `.env.example` with documented placeholders. `MONGO_URI` in
`.env.example` is now a generic placeholder too; the local `.env` still has the old
Mongo password until the user finishes rotating it in Atlas.

## What shipped

- `services/audioCryptoService.js` — AES-256-GCM (authenticated encryption, not just
  confidentiality) `encryptBuffer`/`decryptBuffer`. Layout on disk: `[12-byte IV][16-byte
  auth tag][ciphertext]`. Key comes from `AUDIO_ENCRYPTION_KEY` (64-char hex = 32
  bytes), a new env var.
- `middleware/uploadAudio.js` switched from `diskStorage` to `memoryStorage` — the
  plaintext audio blob now never touches disk unencrypted. `stopConversation`
  encrypts `req.file.buffer` and writes only ciphertext to `backend/storage/audio/`.
- `runSpeechProcessing` (transcription/diarization) and `getConversationAudio`
  (playback/download) both decrypt on demand: the former to a temp plaintext file in
  the OS tmpdir (ffmpeg needs an actual file path), deleted immediately after use,
  same pattern as the existing WAV/JSON cleanup; the latter decrypts straight to a
  buffer and sends it directly, no temp file needed at all for that path.
- Caught a real bug while wiring this up, before it shipped: `stopConversation` still
  referenced `req.file.filename` when kicking off `runSpeechProcessing` — that field
  only exists with `diskStorage`, not `memoryStorage`, so it would have passed
  `undefined` as the filename. Fixed to use the generated encrypted filename instead.

## Verified

- Confirmed the file written to `backend/storage/audio/` is genuinely encrypted, not
  just renamed: read its first bytes and got opaque noise, not the WebM/EBML magic
  header (`1a45dfa3...`) the plaintext file starts with. Size matched exactly
  original + 28 bytes (12-byte IV + 16-byte auth tag), confirming the expected layout.
- Ran the full pipeline end-to-end with real speech through the encrypt/decrypt
  round trip: transcription and diarization both still worked correctly after
  decrypting to a temp file for ffmpeg/whisper.
- Downloaded the audio through the real (now-decrypting) endpoint and byte-compared
  it to the original plaintext file — **identical**, confirming a lossless
  encrypt→store→decrypt→serve round trip, with the correct WebM header restored.
- Confirmed the full audit trail (start/stop/download-audio) still recorded
  correctly through the new encrypted-storage path.
- Checked server logs directly for errors (not just the API responses) — none.
- Deleted all test records/audio afterward.

# Day 10 (part 2) — Kannada: LibreTranslate → NLLB-200

During review the user asked for Kannada — which LibreTranslate genuinely cannot do
(its Argos catalog has no `kn`; that's exactly what crash-looped the container earlier
when `LT_LOAD_ONLY="en,hi,kn"` was tried). Rather than commit a translate feature that
can't do the language the user actually needs, swapped the engine.

## Engine swap: NLLB-200 distilled 600M, local Flask server

- **Model**: `facebook/nllb-200-distilled-600M` — free, open-weight, **ungated** on
  Hugging Face (no account/token needed, unlike pyannote back on Day 4), and covers
  English, Hindi, and Kannada (plus ~200 others addable later by extending one map).
- **Server**: `pyservices/translate_server.py`, a small Flask app exposing the *same*
  `/languages` + `/translate` interface LibreTranslate used — so the Node client
  (`translateService.js`) only changed its comments and env var name
  (`LIBRETRANSLATE_URL` → `TRANSLATE_URL`, old name still honored as a fallback), and
  the controller/UI didn't change at all. Model (~2.4GB) downloads once on first run.
- **Setup**: `npm run setup:translate` (`scripts/setupTranslateTools.js`) creates the
  venv and installs torch/transformers/sentencepiece/flask — all prebuilt cp314
  wheels, no compiler (checked with dry-run first, then a real install, per the Day 4
  lesson).

## Two real Windows walls hit and worked around

1. **`MAX_PATH` overflow installing torch**: torch bundles a license directory nested
   ~10 levels deep; under the long repo path the total exceeded Windows' 260-char
   limit and pip failed with `WinError 206`. Long-path support is disabled on this
   machine (`LongPathsEnabled = 0`) and enabling it needs admin, so instead the
   translate venv lives at a short root (`C:\cpt`, overridable via `TRANSLATE_VENV`)
   rather than inside the repo. Documented in the setup script, since anyone cloning
   on Windows hits the same thing.
2. **cp1252 console encoding**: the first Kannada test "failed" with a
   `UnicodeEncodeError` that was purely Windows' default console codepage being unable
   to *print* Kannada — the translation itself had worked. Re-ran with UTF-8 stdout.
   The Flask server is unaffected (JSON over HTTP, not console prints).

Also noticed Docker Desktop had shut down on its own, taking the (now-superseded)
LibreTranslate container with it — conveniently freeing port 5555 for the NLLB server.
The container/image can be removed entirely at leisure; nothing references it anymore.

# Day 11 — Medication & symptom highlights (post-sprint)

First day of ongoing post-sprint work. Directly serves the original ask: the take-home
sheet should surface "when to take which tablet, symptoms he/she had."

## What shipped

- `services/clinicalExtract.js` — `extractKeyItems(segments)` pulls three categories out
  of the transcript text: **medications** (curated keyword list), **dosage & timing**
  (regex patterns like "twice daily", "after food", "every 8 hours", "one tablet", "for
  five days"), and **symptoms** (keyword list). Deliberately deterministic keyword/regex,
  not an LLM — free, offline, and easy for a doctor to sanity-check. An assist that flags
  candidates, not an authority.
- Backend attaches a computed `keyItems` to session responses (`listConversations` +
  `getConversation`) via a `withKeyItems` helper that uses `toJSON()` (so the speakerRoles
  Map still serializes correctly) and computes on read, so it always reflects the current
  transcript — nothing stored, no schema change.
- Excel export gains a "Key items" summary block at the top (Medications / Dosage & timing
  / Symptoms rows, values merged across columns for room), above the transcript table.
- UI: a "Key items" box on the session view with colour-coded chips (green medications,
  blue timing, red symptoms). Shared `SessionTranscript` component, so it appears both on
  the dashboard's session detail page and the Conversation page's inline list.

# Day 20 — Microservice split, slice 1: speech worker + RabbitMQ + gateway

The heavy AI pipeline (decrypt → whisper → diarize → save) is now a separate
service. On Stop, the API publishes a job to RabbitMQ and returns; a dedicated
worker process (`backend/worker.js`, `npm run worker`) consumes the queue with
`prefetch(1)` — run several workers and the broker load-balances between them.
The pipeline itself was extracted to `services/speechPipeline.js`, shared by
both the worker and the API's fallback path.

**Fallback by design:** if the broker is unreachable, `publishSpeechJob`
returns false and the API runs the pipeline in-process exactly as before — a
broker outage degrades performance, never drops transcripts, and
`docker compose up` stays optional for dev.

**Infrastructure:** `docker-compose.yml` at the repo root brings up RabbitMQ
(healthcheck, management UI on :15672) and an Nginx **API gateway** on :8080
routing `/api/*` → API service and `/` → frontend. The Node services and the
worker stay on the host during dev — they shell out to Windows-native
whisper/ffmpeg binaries and local Python venvs that a Linux container can't
run; containerizing them (Linux tooling builds) is the documented next step,
along with the fuller per-domain service split (auth/patients as separate
services with per-service DBs — explicitly NOT done).

Two real gateway bugs found and fixed during verification: (1) sending
`Connection: upgrade` on every proxied request broke plain HTTP — fixed with
the standard `map $http_upgrade` pattern; (2) Vite binds to localhost only, so
the containerized gateway couldn't reach it via host.docker.internal — fixed
with `server.host: true` in vite.config.js.

## Verified

- Broker down: stopped a session → API logged "RabbitMQ unavailable, falling
  back to in-process speech processing" → transcript completed anyway.
- Broker + worker up: stopped another session → the worker's own log shows
  "[worker] processing/finished session <id>" for exactly that session, the
  transcript completed, and the API logged no fallback — the job genuinely
  travelled through the queue to a separate process.
- `docker compose ps`: rabbitmq healthy, gateway up.
- Gateway: `:8080/api/health` returns the API's health JSON; `:8080/` returns
  the CarePulse frontend (title verified) after the Vite host fix.
- Test data cleaned up afterward.

# Day 21 — Per-domain service split with per-service databases

The monolith is gone. Three services, each its own process with its own
database on the same free Atlas cluster:

| Service              | Port | Database                 | Owns                          |
|----------------------|------|--------------------------|-------------------------------|
| auth-service         | 5001 | carepulse_auth           | Doctor (+ admin passkey login)|
| patient-service      | 5002 | carepulse_patients       | User, Patient, Appointment    |
| conversation-service | 5003 | carepulse_conversations  | ConversationSession, AuditLog |

`config/createService.js` is the shared bootstrap (CORS/JSON/logging/health/
errors) so the three entrypoints stay thin. The speech worker connects to the
conversation service's database — it writes the transcripts that service owns.
Services never read each other's data: identity crosses service boundaries
only via the JWT (shared JWT_SECRET), which is why the split needed no
service-to-service HTTP at all.

**Gateway is now the front door.** nginx routes `/api/auth/*` → 5001,
`/api/conversations*` → 5003, other `/api/*` + `/uploads/*` → 5002, `/` →
frontend. The frontend's default API base changed from :5000 to :8080/api, so
`docker compose up -d` is part of dev startup now. The monolith `server.js`
was deleted rather than left around — two write paths against diverging
databases would have been a split-brain bug factory.

**Migration:** `npm run migrate:service-dbs` copies each collection from the
monolith `carepulse` db into its service's db (upsert by _id, idempotent).
The source db is not modified — it stays as a backup. Ran it: 2 doctors,
2 users, 2 patients, 3 appointments, 3 sessions, 7 audit entries copied,
including the user's real accounts and recordings.

**This is a monorepo split** — one `backend/` package with three entrypoints
sharing models/services code, deployed as separate processes. Deliberate: the
services are genuinely independent at runtime (own process, own port, own DB)
without tripling dependency maintenance. What's still NOT done: containerizing
the services/worker (next: Linux whisper/ffmpeg + python in an image).

## Verified

- All three services healthy on their ports, each logging its own DB name.
- Cross-service JWT: a token issued by auth-service authorized requests on
  patient-service (patient lookup) and conversation-service (start session).
- Broker-down fallback still works in the split conversation-service
  (transcript completed in-process).
- Full flow entirely through the gateway (:8080): start → multipart audio stop
  → RabbitMQ → worker (its log shows the exact session) → transcript readable
  back through the gateway, written to carepulse_conversations.
- Gateway routing spot-checked per domain (distinct, correct responses from
  each service) and the frontend serves through :8080 with its title intact.
- Frontend builds with the new gateway API base. Test data cleaned from all
  three service DBs afterward.
